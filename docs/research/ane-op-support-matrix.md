# ANE Compiler Op Support Matrix

Probed against Apple's private `_ANEInMemoryModel` compiler on macOS (Apple Silicon).
Generated by `cargo run -p ironmill-ane --example ane_op_probe`.

> **Note:** Results may vary by macOS version and chip generation. These results were
> obtained using the `program(1.3)` / `func main<ios18>` MIL text format with
> `[1,C,1,S]` tensor layouts (ANE-native NCHW). Some ops that fail standalone may
> work in specific contexts (e.g., as part of a fused pattern).
>
> **Methodology:** Each op was tested by compiling a minimal single-op MIL program
> via `AneCompiler::compile_mil_text`. Ops marked вњ… were also verified with eval-time
> numerical correctness checks (see `ane_op_eval` вЂ” 59/59 eval-verified ops pass).
> Additional ops were discovered via name fuzzing (see `ane_op_fuzz`). Name fuzzing
> confirmed that failing ops like `neg`, `log`, `scatter` have **no hidden aliases**
> вЂ” they are genuinely unsupported.
>
> **Shape-dependent compilation:** The probe uses `[1,256,1,64]` tensors. Some ops
> that compile at that size fail at smaller sizes (e.g., `[1,32,1,32]`). Ops in the
> unsupported section failed at both sizes. All eval-verified (вњ…) ops were tested at
> `[1,32,1,32]` вЂ” the smaller size confirms they work even with tight tensor budgets.

## Supported Ops (69 confirmed)

> **Data type support (empirically verified via `ane_dtype_probe`):**
> - **Inputs:** only `fp16`, `fp32`, and `bool` accepted as function inputs
> - **INT8 storage:** `cast fp16в†’int8` вњ…, `cast int8в†’fp16` вњ…, `dequantize(int8)` вњ…

### INT8 KV Cache Storage вЂ” вњ… VERIFIED (2Г— bandwidth reduction)
The ANE supports INT8 as a **storage format** with fp16 arithmetic:
- **Quantize on ANE:** `mul` в†’ `round` в†’ `clip` в†’ `cast fp16в†’int8` (all eval-verified)
- **Dequantize on ANE:** `cast int8в†’fp16` в†’ `mul` в†’ `add` (all eval-verified)
- **Bandwidth saving:** 1 byte/element vs 2 bytes/element (2Г— reduction)
- **Full pipeline verified:** `test_turboquant_int8_cache_pipeline` runs
  RMSNorm в†’ quantize в†’ dequantize в†’ attention dot product in a single
  ANE sub-program, matching CPU reference within max_err=0.014.

### INT4 KV Cache вЂ” вќЊ NOT FEASIBLE via MIL text
INT4/UINT4 are comprehensively rejected by the ANE compiler across all paths:
inputs, casts, arithmetic, matmul, conv, dequantize. Verified by `ane_dtype_probe`.
The 4Г— bandwidth reduction claimed in `turboquant-ane-orion.md` is **not achievable**
through the MIL text compilation path. maderix/ANE's INT4 results may use a
lower-level weight descriptor API that bypasses MIL compilation.

### LUT Dequantization (static weights) вЂ” вњ… NOT BLOCKED
`constexpr_lut_to_dense` is a **compile-time constant-folding op**, not a runtime
`gather`. The ANE compiler expands the LUT into dense weights during compilation.
This is the same mechanism used by `constexpr_affine_dequantize` for INT8 weights
(confirmed working by maderix/ANE). Therefore:
- **Static weight palettization/PolarQuant**: not affected by `gather` failure.
  The compiler bakes the LUT expansion into the weight blob at compile time.
- **Runtime LUT dequantization** (e.g., dynamically quantized activations): would
  require a runtime `gather`, which is unsupported. Workarounds:
  - Use arithmetic dequantization (`mul` + `add`) instead of LUT-based
  - Decompose into `select` chains for small codebooks (2-4 entries)
  - Fall back to CPU at sub-program boundaries

See also: CoreML only supports LUT sizes `{2, 4, 16, 64, 256}` в†’ `n_bits в€€ {1, 2, 4, 6, 8}`.
3-bit (8 entries) is rejected. See `KNOWN_ISSUES.md`.

### Cache Mutation вЂ” вќЊ REQUIRES DECOMPOSITION
All `scatter` variants are rejected by the ANE compiler (compile-probed, 3 variants).
Name fuzzing confirmed no aliases exist. Four decomposition paths are available,
with `concat` and `select` eval-verified:

| Strategy | ANE-only? | Position type | Cost | Eval status |
|---|---|---|---|---|
| `concat` (append) | вњ… | Append-only | O(1) | вњ… eval verified (max_err=0.025) |
| `select` + mask | вњ… | Any position | O(max_seq) | вњ… eval verified (exact) |
| Arithmetic mask (`mul`/`add`) | вњ… | Any position | O(max_seq) | Ops individually verified |
| CPU interception | Hybrid | Any position | O(1) memcpy | N/A (host code) |

For autoregressive decoding, **CPU interception at sub-program boundaries** is the
pragmatic choice. The ironmill-ane `predict()` loop already copies tensors between
sub-programs in Rust. Adding a partial-write API (`AneTensor::write_f16_at`) would
enable O(1) cache updates. For append-only patterns (e.g., prefill), pure-ANE
`slice_by_index` + `concat` works without CPU involvement.

## Notes

### Compile-time vs runtime ops
`constexpr_*` ops (e.g., `constexpr_lut_to_dense`, `constexpr_affine_dequantize`)
are **compile-time constant-folding ops**. The ANE compiler expands them into dense
weight blobs during compilation. They are not executed at runtime, so runtime op
support (e.g., `gather` failing) does not affect them. This is confirmed by:
- maderix/ANE using `constexpr_affine_dequantize` for INT8 weights in production
- ironmill's proto converter treating `constexpr_*` as constant-bearing nodes
- The palettization and PolarQuant passes both emit `constexpr_lut_to_dense`

### Ops confirmed by maderix/ANE project
The [maderix/ANE](https://github.com/maderix/ANE) project independently confirms
these ops work in production ANE training kernels:
- `conv`, `matmul`, `add`, `mul`, `sub`, `sigmoid`, `softmax`, `reshape`,
  `transpose`, `reduce_sum`, `pow`, `concat`, `slice_by_size`
- `constexpr_affine_dequantize` for INT8 weights
- `quantize`/`dequantize` for INT8 activation caching (1.88Г— throughput)

### Common decomposition patterns
Many "unsupported" ops have simple decompositions into supported ops:
- `neg(x)` в†’ `sub(0, x)` or `mul(x, -1)`
- `rsqrt(x)` в†’ `pow(x, -0.5)`
- `inverse(x)` в†’ `real_div(1, x)`
- `logical_and(a, b)` в†’ `cast(mul(cast(a, fp16), cast(b, fp16)), bool)`
- `sinh(x)` в†’ `(exp(x) - exp(neg(x))) / 2` в†’ `real_div(sub(exp(x), exp(sub(0,x))), 2)`
- Runtime quantize в†’ `mul` в†’ `round` в†’ `clip` в†’ `cast`

### Shape constraints
Some ops may fail due to shape constraints rather than being fundamentally
unsupported. The ANE has specific alignment requirements:
- Input channels should be aligned to 32
- Conv kernels в‰¤ 16Г—16
- Tensor layout must be `[1, C, H, W]` (NCHW)
- Reduction outputs with small spatial dims (`[1,C,1,1]`) have IOSurface
  alignment issues when read directly вЂ” broadcast back to full size via
  `tile` before reading output tensors

### Clarifications from cross-project analysis

**`sin` / `cos` вЂ” NOT used as MIL compute ops by maderix:**
Inspection of `maderix/ANE` `mil_dynamic.h` reveals that RoPE uses
*precomputed* sin/cos tables loaded as `const()` from BLOBFILEs, then applied
via `mul` and `add`. The `sin()` and `cos()` MIL compute ops are never invoked.
ironmill's probe correctly shows these ops as unsupported вЂ” the discrepancy
was a misattribution. A RoPE pattern test using precomputed tables has been
added to `ane_op_eval.rs` to verify this works on ANE.

**`quantize` вЂ” standalone vs fused context:**
ironmill's probe rejects standalone `quantize`. maderix's `ane_int8_bench.m`
uses `quantize` inside a fused `conv в†’ quantize в†’ dequantize` pipeline. A
fused-context test matching maderix's pattern has been added to
`ane_op_eval.rs` to determine whether the ANE compiler accepts `quantize`
only in specific pipeline contexts.

**`pad` вЂ” previously untested:**
Orion includes `ORION_OP_PAD` in its graph IR but never documents its ANE
verification status. A `pad` (constant mode) eval test has been added to
`ane_op_eval.rs`.

**New tests added:** `ane_op_eval.rs` now includes 4 cross-project
verification tests:
- `pad constant` вЂ” вњ… **PASS** вЂ” pad works on ANE (fp16 precision, max_err=0.025)
- `quantize standalone` вЂ” вњ… **PASS** вЂ” standalone quantizeв†’dequantize works
  (contradicts previous probe result вЂ” may have been fixed in macOS update)
- `quantize fused convв†’qв†’dq` вЂ” compiles but eval fails with `status=0x1d`
  (IOSurface sizing issue with multi-input conv pattern, not a quantize problem)
- `RoPE pattern (precomp sin/cos)` вЂ” вњ… **PASS** вЂ” confirms maderix's actual
  pattern works (precomputed tables via `mul`+`add`, not `sin()`/`cos()` ops)

**Newly confirmed ANE-supported ops:**
- **`pad`** (constant mode) вЂ” eval-verified, max_err=0.025
- **`quantize`** (standalone, with dequantize) вЂ” eval-verified, max_err=0.0
- **`rsqrt`** (with epsilon parameter) вЂ” eval-verified. ANE requires the
  `epsilon` param; without it compilation fails. This was previously listed
  as unsupported вЂ” it was a MIL formatting issue, not a hardware limitation.
- **`log`** (with epsilon parameter) вЂ” eval-verified, max_err=0.005. Same
  epsilon requirement as rsqrt. No logarithm function was thought to exist
  on ANE вЂ” this is a significant discovery.
- **`inverse`** (with epsilon parameter) вЂ” eval-verified, max_err=0.001.
  Same epsilon pattern. Previously decomposed to `real_div(1, x)`.

**ANE epsilon requirement:** The ANE compiler requires an explicit `epsilon`
parameter for `rsqrt`, `log`, and `inverse` вЂ” even though standard MIL treats
it as optional. Without epsilon, these ops fail to compile. This is the single
most impactful finding from the cross-project investigation, recovering 3 ops
that were believed to be hardware-unsupported.

**Failure investigation results (12 remaining compile failures):**
An exhaustive probe tested alternative op names, parameter formats, dtypes
(fp16/fp32), and tensor sizes for each failing op:

- **`neg`** вЂ” genuinely unsupported (`neg`, `negate` both fail). Decompose
  via `mul(x, -1)` or `sub(0, x)`.
- **`log`** вЂ” genuinely unsupported (`log`, `ln`, `log_e`, `log2` all fail
  **without epsilon**). **Works with `epsilon` parameter** вЂ” see above.
- **`inverse`** вЂ” genuinely unsupported **without epsilon**. **Works with
  `epsilon` parameter** вЂ” see above.
  Decompose via `real_div(1, x)` if epsilon is unavailable.
- **`sin`, `cos`, `tan`** вЂ” genuinely unsupported across all dtypes and
  tensor sizes. Only `atan` (unary) works among trig functions.
- **`asin`, `acos`** вЂ” genuinely unsupported. `atan2` also unsupported.
- **`sinh`, `cosh`** вЂ” genuinely unsupported. Only `tanh` works (likely a
  dedicated hardware unit on ANE).
- **`mod`** вЂ” genuinely unsupported (`mod`, `fmod`, `remainder`, `floor_mod`
  all fail). Decompose via `sub(x, mul(floor_div(x,y), y))`.
- **`logical_and`, `logical_or`, `logical_xor`** вЂ” the ops themselves are
  unsupported, but decomposition via fp16 arithmetic works:
  `and` в†’ `mul(cast(a,fp16), cast(b,fp16))`,
  `or` в†’ `maximum(cast(a,fp16), cast(b,fp16))`.
